Этап А: Загрузка (Upload)
    User: Драг-н-дроп видео.
    FastAPI: Сохраняет в MinIO (Bucket: raw). Создает запись в БД со статусом UPLOADED.
    Response: Возвращает ID видео и превьюшку.
    UI: Показывает видео в списке: "Загружено. Выберите действие".

Этап Б: Запуск задачи (Trigger)
    User: Выбирает в UI алгоритм (напр., "YOLOv8 Nano" или "OpenCV Edge Detect") и жмет "Обработать".
    Frontend: Шлет POST запрос: /api/v1/process/{video_id} с телом {"algorithm": "yolo_v8", "conf": 0.5}.
    FastAPI:
        Проверяет, что видео существует.
        Меняет статус в БД на QUEUED.
        Кидает задачу в Redis: {"video_id": 1, "algo": "yolo", "path": "s3://..."}.
    Worker: Подхватывает задачу, меняет статус на PROCESSING, и погнали.
Итоговая "Крепкая" Архитектура v2.0
Теперь у нас полный набор.
Список контейнеров (Services):
    Traefik / Nginx (Reverse Proxy) — входная точка.
    Frontend (Nginx + React build) — статика.
    Backend (FastAPI) — бизнес-логика, API.
    Database (PostgreSQL) — метаданные, юзеры, статусы задач.
    S3 Storage (MinIO) — хранение сырых и готовых видео.
    Broker (Redis) — очередь задач.
    AI Worker (Python + YOLO) — потребитель задач.
Почему это круто?
    Stateless: Бэкенд и Воркер не хранят состояние. Если они упадут — данные в MinIO и Postgres в безопасности.
    Decoupling: Ты можешь добавить кнопку "Отменить обработку", и бэкенд просто удалит задачу из Redis, даже не трогая воркера.
    Flexibility: Ты можешь добавить второй тип воркера (например, для аудио), и он будет слушать другую очередь в том же Redis.
